- Suggest solution: "Try again", "Check connection"

- **Progressive** - Use present continuous for actions

## General Translation Best Practices

### 1. Context is Everything

Always consider:
- **Who is reading?** (User, admin, new visitor)
- **Where does it appear?** (Button, error message, email)
- **What emotion should it convey?** (Celebration, concern, information)

**Example:**
- Button: "Save" â†’ Short, action-oriented
- Success message: "Lighter saved successfully!" â†’ Celebratory
- Empty state: "No lighters saved yet" â†’ Encouraging

### 2. Consistency Across the Platform

**Use the same translation for:**
- Repeated UI elements (buttons, navigation)
- Status messages (Loading..., Success!, Error)
- Standard actions (Save, Cancel, Delete)

**Keep a translation memory** or glossary for your language.

### 3. Handle Variables Properly

**Variables use curly braces:** `{variable}`

âœ… **CORRECT:**
- English: `"Welcome back, {name}!"`
- French: `"ContentÂ·e de te revoir, {name} !"`
- German: `"Willkommen zurÃ¼ck, {name}!"`
- Spanish: `"Â¡Bienvenide de vuelta, {name}!"`

âŒ **INCORRECT:**
- Don't translate: `"Welcome back, {nombre}!"`
- Don't remove: `"Welcome back!"`
- Don't modify syntax: `"Welcome back, {name}!"`

**Variable types:**
- `{name}` - User names
- `{count}` - Numbers
- `{date}` - Dates
- `{amount}` - Currency
- `{error}` - Error messages

### 4. Plural Handling

Many languages have complex plural rules. Check how your language handles:

**English pattern:**
- `{count} sticker` (count = 1)
- `{count} stickers` (count â‰  1)

**French pattern:**
- `{count} sticker` (count = 0 or 1)
- `{count} stickers` (count > 1)

**Use proper plural keys when provided:**
```typescript
'order.sticker_singular': '{count} sticker',
'order.sticker_plural': '{count} stickers',
```

### 5. Technical Terms

**When to translate:**
- UI elements: "Button", "Menu", "Search"
- Common actions: "Save", "Delete", "Edit"
- User-facing features

**When NOT to translate:**
- Brand names: "LightMyFire", "LightSaver"
- Product names: "Sticker", "PIN"
- Technical standards: "QR code", "URL", "API"
- File formats: "PDF", "JPG"

**When in doubt:** Keep it in English if widely understood by your target audience.

### 6. Length Constraints

**Be mindful of space:**
- Buttons: Keep short (2-4 words)
- Navigation: Concise labels
- Mobile screens: Even shorter

**German tends to be longer** - use compound words wisely
**Spanish can be longer** - prefer concise options
**French middle ground** - optimize without losing meaning

### 7. Cultural Sensitivity

**Avoid:**
- Religious references (unless brand-relevant)
- Political statements
- Offensive humor or stereotypes
- Cultural assumptions

**Embrace:**
- Universal human emotions (joy, curiosity, belonging)
- Shared values (creativity, community, sustainability)
- Local linguistic flavor (idioms, expressions)

## Technical Guidelines

### File Structure

**Location:** `/locales/{language}.ts`

**Format:**
```typescript
export default {
  'key.name': 'Translation text',
  'another.key': 'More text',
} as const;
```

**Rules:**
- Single quotes for strings
- Comma after each entry
- Dot notation for nested keys: `home.hero.title`
- `as const` at the end for TypeScript

### Key Naming Convention

Keys follow this pattern: `{section}.{subsection}.{element}`

**Examples:**
- `nav.how_it_works` - Navigation > How It Works
- `home.hero.title` - Home page > Hero section > Title
- `error.network` - Errors > Network error
- `order.payment.total` - Orders > Payment > Total

**Never modify key names** - only translate the values!

### Special Characters

**Escaping:**
- Single quotes: `'Don\'t throw it away'`
- Curly braces: Keep as-is for variables
- Line breaks: `\n` for new line
- HTML: Can include `<strong>`, `<br>`, links

**Example:**
```typescript
'welcome.message': 'Welcome to <strong>LightMyFire</strong>!\nYour lighter\'s journey begins now.',
```

### Testing Your Translations

**Before committing:**

1. **Syntax check:** Ensure file compiles
   ```bash
   npm run build
   ```

2. **Visual check:** View in app
   - Check text fits in buttons/UI
   - Verify line breaks make sense
   - Test on mobile screens

3. **Proofread:**
   - Read aloud
   - Check for typos
   - Verify consistency

4. **Variables:** Ensure they work
   - Test with different values
   - Check plural forms
